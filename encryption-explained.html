The two kinds of keys, in plain words.

You asked a sharp question: is there one key, or two? Here's the whole thing with the jargon stripped out — and which kind this scam actually uses.

1First, what a "key" even is

Encryption = scrambling something so it looks like nonsense. Decryption = unscrambling it back. To do either, you need a key — just think of it as a password.

Yes — there are proper maths "recipes" (algorithms) that do this scrambling. The famous ones have names like AES and RSA. You don't need to know the maths — only how the keys are used. And there are exactly two families, based on your question:

2The two families

🔑🚪
A · One key, same both ways
nickname: "symmetric"

Like your house key. The same key locks the door and unlocks it. Whoever holds that one key can do both.

The catch: you must keep that key secret — and if you want someone else to unscramble your message, you have to hand them the very same key, safely.

Real algorithm: AES. It's fast, so it's used for scrambling big things.

👉 This is what the scam page uses.
📬🔐
B · Two keys, a matched pair
nickname: "asymmetric" / "public-key"

Like a mailbox with a slot. Anyone can drop a letter in through the slot — that's the "public" key, you can share it with the whole world. But only you have the little key that opens the box to take letters out — that's the "private" key, you never share it.

So one key locks, and a different key unlocks. They're made together as a pair.

Real algorithm: RSA. It's what the padlock 🔒 in your browser (HTTPS) uses to get started.

✗ Not used by this scam.
Direct answer to your a & b

(a) Same key for both → yes, that exists — it's Family A (symmetric), e.g. AES.  (b) Lock with one, unlock with a different one → yes, that exists too — it's Family B (asymmetric), e.g. RSA.

4So which does the scam use? — Family A

The scam uses the one-key kind (AES). The scammer picked a single password, used it to scramble the hidden "black page," and the visitor's browser uses that same password to unscramble it.

Why that let us crack it open

Because it's the one-key kind, and that one key had to be inside the page for the browser to use it — so we just read it off and unscrambled everything ourselves.  (With the two-key kind, even finding the locking key wouldn't help — you'd still need the separate, secret unlocking key that never leaves the owner.)

5How it actually runs — broken all the way apart

Q1. Do I feed encrypted data at the start, or does encryption happen later?

  • The data is already encrypted, ahead of time. The scammer scrambled the hidden page once, on their own computer, before putting anything online.
  • The web page (in your browser) never encrypts anything. Its only job is to unscramble — decrypt.
  • So the order is: ready-made scrambled data sits on the server → your browser downloads it → your browser unscrambles it.
In one line

Encryption happened earlier, offline, once. The browser only ever decrypts — it never encrypts.

Q2. Is the key inside crypto-js? Or is crypto-js just the tool that runs the decryption?

  • crypto-js does NOT contain the key. It's just the engine — generic code that knows how to scramble and unscramble. Everyone downloads the exact same crypto-js; it's identical on every website in the world.
  • The key is separate. It's a plain password written in the page's own script (const PASSPHRASE = "…") — not inside crypto-js.
  • At run time, the page hands the key to crypto-js: "here's the scrambled text, here's the password — unscramble it." crypto-js does the turning and gives back the original.
In one line

crypto-js = the machine. The key = the password you drop into the machine. Two separate things — crypto-js is only used to run the decryption; it never holds your key.

Q3. The whole run, start → finish

  • (Before anyone visits — offline) Scammer scrambles the hidden page with a password → gets a blob of scrambled text.
  • Scammer uploads that blob to the vault server, and — inside the white page — writes the password and loads the crypto-js tool.
  • You open the white page → crypto-js loads, the password sits in the page, waiting.
  • Your mouse moves → the page downloads the scrambled blob from the vault.
  • The page tells crypto-js: "unscramble THIS blob using THIS password."
  • crypto-js hands back the original hidden-page HTML.
  • The page shows it. No encryption ever happened in your browser — only that one un-scramble step.
The whole thing in one breath

The scrambling was done once, in advance. Your browser only ever unscrambles — using a password that rides inside the page and a generic tool (crypto-js) that does the turning.

6About that scary command earlier — ignore it

Earlier I showed an openssl … line and it looked like gibberish. Forget it — you don't need it to understand any of this.

All that command was: a free tool that already comes with most computers. You feed it the scrambled text plus the password, and it hands back the original. That's it. I only showed it to prove the scrambling here is an ordinary, standard method — not some custom hacker magic. The idea is all that matters:

scrambled text + password → original text.
